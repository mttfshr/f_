# f_mobius

**Status:** Specced, not yet built
**Lifecycle stage:** ideas/ — ready to graduate to .specify/ when build begins

---

## Concept

A UV-space processor that applies a Möbius transformation to the sampling coordinates of an incoming texture. Every pixel is remapped to a new source location — the texture itself is unchanged, but where each pixel samples from moves according to the transformation.

Möbius transformations map circles to circles (lines are circles of infinite radius) and preserve angles between curves. The result feels smooth and geometrically coherent even at extreme settings — no pinching, no shearing. The transformation is defined by:

```
f(z) = (az + b) / (cz + d)
```

where z is the pixel coordinate treated as a complex number (x + iy), and a, b, c, d are complex constants derived from the performer controls.

**Position in a chain:** Processor. Takes texture in, outputs warped texture. Sits between a source (f_chladni, video, noise) and any downstream processors or the fisheye/circular screen display layer. Because it warps UV space rather than pixel values, it composes cleanly with any other UV-space processor.

## Parameters

| Param | Range | Default | Description |
|-------|-------|---------|-------------|
| `cx` | 0.0–1.0 | 0.5 | Center point X — anchor of the transformation in UV space |
| `cy` | 0.0–1.0 | 0.5 | Center point Y — anchor of the transformation in UV space |
| `rotate` | 0.0–1.0 | 0.0 | Rotation angle around center (mapped to 0–2π in codebox) |
| `zoom` | 0.0–1.0 | 0.5 | Scale factor (log-mapped: 0.5 = unity, 0 = 0.1×, 1 = 10×) |
| `invert` | 0.0–1.0 | 0.0 | Inversion blend: 0 = identity/rotation/zoom only, 1 = full 1/z |
| `bypass` | 0/1 | 0 | Standard bypass |

**Center point (`cx`, `cy`):** Intended to be driven by an xy encoder. The transformation is computed relative to this point — it stays fixed while everything around it warps. At center (0.5, 0.5) the transformation is symmetric. Off-center it becomes asymmetric in interesting ways.

**Invert:** This is the unique control. At 0, f_mobius is a smooth rotation and zoom tool — useful but familiar. As invert increases, the inside and outside of the transformation begin to trade places. Near-center content expands toward the edges; edge content contracts toward center. At 1, the transformation is pure inversion (1/z): the center goes to infinity and the edges come in. The musically interesting region is 0.2–0.8 — partial inversion combined with rotation produces **loxodromic** transformations, the hyperbolic motion territory where content appears to spiral inward or outward continuously.

## Codebox Structure

Complex arithmetic uses vec2 as the number type (real, imaginary).

```
// Complex helpers
vec2 cmul(vec2 a, vec2 b) = (a.x*b.x - a.y*b.y, a.x*b.y + a.y*b.x)
vec2 cdiv(vec2 a, vec2 b) = cmul(a, conj(b)) / dot(b, b)

// Signal flow:
1. Map norm (UV) → complex plane centered at (cx, cy)
2. Apply rotation + zoom to get a, b, c, d
3. Compute identity result and inversion result separately
4. Blend between them by invert param
5. Map result back to UV space
6. sample(in1, uv_out)
7. mix(effect_out, sample(in1, norm), bypass)
```

**Separation of identity and inversion paths** (step 4) keeps the blend clean — no discontinuities as invert moves from 0 to 1. The rotation and zoom parameters affect both paths consistently.

**Zoom mapping:** `actual_zoom = pow(10.0, (zoom - 0.5) * 2.0)` — gives 0.01× at zoom=0, 1× at zoom=0.5, 100× at zoom=1. May need narrowing empirically; start with `pow(10.0, (zoom - 0.5) * 1.0)` for a tighter range (0.1× to 10×).

## Relation to Other Bpatchers

**Enables downstream:**
- `f_stereo` (planned) — stereographic projection; Möbius transformations *are* sphere rotations in stereographic projection, so f_mobius + f_stereo together produce honest sphere rotation
- `f_poincare` (planned) — Poincaré disk hyperbolic tiling; the disk automorphisms are the Möbius transformations that keep the unit disk inside itself; f_mobius with constrained parameters is the motion model

**Shares math with:**
- Apollonian gasket — generated by repeated circle inversions; the inversion operation in f_mobius is the same primitive
- Non-Euclidean geometry work generally — Möbius transformations are the symmetry group of the Riemann sphere

**Works well with:**
- f_chladni — Möbius-warped Chladni patterns; nodal rings stay rings, but their layout inverts and rotates in interesting ways
- f_droste — both are UV-space processors; order matters and both orders are worth exploring
- Any circular/radially-symmetric source — the circle-preserving property of Möbius transformations respects radial symmetry

## Open Questions

- **Zoom range:** `pow(10, ±1)` = 0.1–10× is a reasonable starting range; tune empirically for what's performable
- **Edge behavior:** What happens at the singularity (where the denominator → 0)? Needs a guard — clamp or modulo UV before sampling. Mirror edges is the natural choice to match the existing spherical-surface illusion approach.
- **Loxodromic sweet spot:** The invert range 0.2–0.8 with rotation is the most interesting territory — worth a performance test session to find the expressive range before locking defaults
- **Second xy encoder:** Could `rotate` and `zoom` also be driven by a second xy encoder (rotate on x, zoom on y)? Would make the full transformation performable with two hands on two encoders. Design decision for build time.
