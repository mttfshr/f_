# Implementation Plan: f_vf_fieldmap

**Date:** 2026-06-07
**Spec:** `.specify/f_vf_fieldmap/spec.md`

## Summary

f_vf_fieldmap is a scalar-to-vecfield converter. One `jit.gl.pix @type float32` codebox reads luminance from an input texture, computes a central difference gradient at each pixel (4 samples), encodes the result as an f_vecfield RG texture (0.5 = zero vector), and outputs it. Two params: `strength` (scales gradient magnitude) and `scale` (central difference neighbor distance). Processor archetype — requires input; vs_black fallback produces neutral field.

This is the simplest f_vf_ patch to date: no analytic math, no fixed-point geometry, no modulation inlets. Complexity lives in the codebox sampling pattern and the `vec()` syntax risk (needs scratch verification before build).

## Architecture Decisions

### ADR 1: Central difference, 4 samples, luminance input

**Context:** Several finite difference schemes were considered (forward, central, 5-point stencil). Input channel selection also open.

**Decision:** Central difference — `grad.x = luma(x+h, y) - luma(x-h, y)`, same for Y. Luminance composite (Rec. 601: 0.299R + 0.587G + 0.114B) as the scalar value. 4 samples total.

**Rationale:** Central difference is symmetric — gradient is correctly centered at the sample pixel, correct zero-crossing at flat regions (encoded output exactly 0.5). Forward difference has a half-pixel offset. 5-point stencil adds diagonal samples for marginal quality improvement not worth the cost. Luminance handles the expected grayscale-dominant inputs (jit.gl.bfg, f_grain, f_stipple) while correctly handling an RGB source if one is patched in.

**Consequences:**
- 4 texture samples per output pixel. Cheap — no branching, no iterative loops.
- Output is exactly 0.5/0.5 on flat input regions. Correct, expected, testable.
- Luma weights hardcoded in codebox — no channel selector param needed.

---

### ADR 2: `scale` param controls neighbor sample distance

**Context:** The central difference step size `h` is an expressive parameter, not just a numerical detail. Small `h` captures fine gradient structure; large `h` smooths over fine detail and extracts coarse structure.

**Decision:** Expose as `scale` param (performer-friendly label). Range 0.001–0.05 normalized UV coords. Default 0.004 (~2px at 480p).

**Rationale:** The name `scale` matches the performer's experience — "turn up scale to get coarser structure." `step` is technically accurate but not intuitive in a live context. Making it a param rather than a constant adds expressive value: a jit.gl.bfg Perlin input with high `scale` produces a smooth low-frequency field; with low `scale` it tracks fine noise structure tightly.

**Consequences:**
- Range upper bound 0.05 = ~24px at 480p. Above this, gradient becomes very coarse and may produce near-zero output with smooth inputs. Not harmful — just less useful.
- Lower bound 0.001 approaching sub-pixel; behavior is well-defined (samples are fractionally interpolated by GPU) but gradient becomes noisy. Minimum validated in scratch.

---

### ADR 3: `strength` default 4.0, range 0–10

**Context:** Raw central difference output for typical jit.gl.bfg Perlin (contrast ~0.3–0.5 of full range) produces gradient magnitudes well below ±0.5 normalized. At `strength = 1.0`, the encoded f_vecfield barely deviates from 0.5, giving consumers almost nothing to work with at defaults.

**Decision:** Default `strength = 4.0`. Puts typical Perlin gradient into useful f_vecfield range without saturation. Range 0.0–10.0 allows user to calibrate for high-contrast inputs (where lower strength prevents clipping) or low-contrast inputs (where higher strength boosts weak gradients).

**Consequences:**
- At `strength = 4.0` with typical jit.gl.bfg output, expect RG values in roughly 0.3–0.7 range — usable dynamic range without hard clamping.
- Saturation (clamp to 0/1) is valid — just loses gradient direction detail at extremes. Not harmful; field is still a valid f_vecfield.
- Calibrate in scratch patch before committing default to `definition.py`.

---

### ADR 4: Processor archetype — single inlet, vs_black fallback

**Context:** f_vf_fieldmap requires a source texture. It has no self-generating mode.

**Decision:** Processor archetype. Inlet 0 carries both the source texture and control messages (standard `routepass jit_gl_texture jit_matrix` pattern). vs_inState on the texture branch; vs_black fallback produces neutral field (all 0.5) when unconnected.

**Rationale:** Consistent with vsynth-bpatcher processor convention. vs_black = 0.5 decoded = zero gradient — the correct neutral behavior for an unconnected source.

**Consequences:**
- Bypass outputs neutral field (all 0.5) — same as unconnected. No source passthrough on bypass (patch produces a vecfield, not a visual texture — passing through the source on bypass would be type-incorrect).
- No dual-mode — patch has one job.

---

### ADR 5: `vec()` syntax for UV offset — verify in scratch

**Context:** The codebox pseudocode uses `sample(in0, norm + vec(h, 0))` to offset the sample coordinate. `vec()` is used throughout jit.gl.pix gen codeboxes but the jit-gen-codebox skill flags that some vector construction forms fail silently or hard-error on GPU.

**Decision:** Verify `vec(h, 0.0)` compiles and produces correct output in a scratch codebox before writing `definition.py`. If `vec()` fails, fall back to explicit coordinate construction: `sample(in0, vec(norm.x + h, norm.y))`.

**Consequences:**
- Scratch verification is a required gate before Phase 2. This is the primary technical risk.
- The fallback is clean and equivalent — no impact on output quality.

## Implementation Phases

### Phase 1: Codebox in scratch patch

Create `fieldmap-scratch.maxpat`. Write codebox implementing:
- Luminance from `in0`: `luma = 0.299*r + 0.587*g + 0.114*b`
- 4 central difference samples using `scale` param as `h`
- Gradient scaled by `strength`
- f_vecfield encoding: `clamp(grad * 0.5 + 0.5, 0, 1)`, B=0.5, A=1.0
- Bypass: output constant 0.5/0.5/0.5/1.0

**Key verification tasks:**
- `vec(h, 0.0)` syntax: does it compile and produce correct UV offset? If not, switch to explicit form.
- Flat input → exactly 0.5/0.5 output (verify in jit.pwindow with solid color source)
- jit.gl.bfg Perlin → visible RG gradient field in jit.pwindow
- Patch into vs_displacement: smooth displacement following noise gradient contours
- Calibrate `strength` default: adjust from 4.0 if needed based on actual jit.gl.bfg contrast
- `scale` low/high produces noticeably different gradient coarseness

**Exit criteria:** Block 1 verification passes. Defaults confirmed.

### Phase 2: definition.py + build

Write `.specify/f_vf_fieldmap/definition.py` using confirmed codebox. Run `tools/build_patcher.py`. Verify generated JSON.

**Param layout:** Row 1: strength, scale. Row 2: bypass (right-aligned in header per convention).
**Presentation height:** Standard 2-param height (same as f_vf_vortex minus mod inlets — check existing for reference).

**Exit criteria:** Block 2 verification passes. Patcher loads in Vsynth.

### Phase 3: Integration with f_caustic

Patch `jit.gl.bfg → f_vf_fieldmap → f_caustic`. Verify caustic bright bands accumulate along noise ridge lines. Try with Perlin, Voronoi, and fractal noise types to confirm the patch works with varied inputs.

**Exit criteria:** Block 3 verification passes. Three-patch signal chain produces compelling output.

Update `README.md` to move f_vf_fieldmap from Planned to Working in the f_vf_ family table. Update `docs/f-reference/f_vecfield_type.md` f_vf_fieldmap row from Planned to Complete.
