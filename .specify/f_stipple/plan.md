# Implementation Plan: f_stipple

**Date:** 2026-05-30
**Spec:** `.specify/f_stipple/spec.md`

---

## Summary

f_stipple is a 2D hash field stipple generator/processor for Vsynth — a single `jit.gl.pix` with a gen codebox, texture+control inlet (routepass pattern). The codebox rotates screen UV into a local `along`/`across` frame, evaluates two hash functions (sin-based for anisotropic lines, arithmetic for isotropic grain), blends them by `anisotropy`, and thresholds against `threshold` with `softness`. Auto-detects source vs processor mode from inlet 0. The patcher is a thin wrapper — the work is almost entirely GLSL.

## Architecture Decisions

### ADR 1: Source/processor mode detected via `vs_inState`, not in-shader

**Context:** f_stipple has two distinct behaviors depending on whether an upstream texture is present. Several detection approaches were considered before inspecting Vsynth source.

**Decision:** Use `vs_inState` — a Vsynth abstraction that detects inlet connection state via draw-callback timing (not alpha or luma). Wire: `routepass out0 → vs_inState → pix in0`. `vs_inState` outlet 0 passes the real texture or `vs_black` fallback; outlet 1 emits a 0/1 int that drives `prepend param src_mode → pix in0`. The codebox reads `Param src_mode` to select source vs processor path. `src_mode` is never exposed as a user-facing param.

**Rationale:** This is how Vsynth's own dual-mode modules work (confirmed by reading `vs_inState.maxpat` and `vs_displacement.maxpat`). Alpha and luma detection are fragile — Vsynth textures may have variable alpha. `vs_inState` uses timing of draw callbacks, which is reliable and context-aware.

**Alternatives considered:**
- Alpha channel check (`sample(in1,norm).a > 0.0`): fragile, some textures have alpha=0 legitimately
- Luma check: slightly more robust but still wrong — a black input texture would be misdetected as no connection
- Explicit `src_mode` param exposed to user: adds a param that the user shouldn't need to set manually

**Consequences:**
- Patcher must include `vs_inState` object and its two outlet wirings
- `src_mode` is a hidden Param in the codebox — present in codebox but absent from UI, route, and parameters block
- Source bypass outputs black; processor bypass passes input through unchanged

---

### ADR 2: Two hash functions blended by anisotropy

**Context:** The expressive range of f_stipple spans directional parallel lines (high anisotropy) to isotropic grain (low anisotropy). These have different optimal hash implementations.

**Decision:**
- High anisotropy end: sin-based hash on `along` axis only — `sin(along * freq * k) * 43758.5453 - floor(...)`. Periodic, directional, tiles cleanly.
- Low anisotropy end: arithmetic hash combining both axes — `sin((along * p1 + across * p2) * freq) * 43758.5453`. Isotropic, no tiling requirement.
- `anisotropy` crossfades: `hash_field = mix(h_iso, h_parallel, anisotropy)`

**Rationale:** Single hash function cannot span both poles cleanly. The crossfade is cheap and produces the structured-grain midrange that is the primary expressive zone.

**Alternatives considered:**
- Single 2D hash for everything: loses the clean line structure at high anisotropy
- `anisotropy` as axis weighting inside one hash: produces subtle variation, not the line↔grain range

**Consequences:**
- Two hash evaluations per pixel — negligible GPU cost
- The 2D combination in `h_iso` is expected to break the diagonal seam artifact from f_masonry's 1D hash. Verify in Phase 1.

---

### ADR 3: Along/across coordinate frame with angle as load-bearing axis

**Context:** Hash orientation and drift direction should both operate in the stipple's own coordinate frame, not screen space.

**Decision:** Rotate UV by `angle` to get `along`/`across`. Both hash evaluation and speed offsets use this frame:
```
theta  = angle * (PI / 180.0)
along  = norm.x * cos(theta) + norm.y * sin(theta)
across = -norm.x * sin(theta) + norm.y * cos(theta)
hash evaluated at (along + along_speed * time, across + across_speed * time)
```

**Rationale:** Makes `along_speed`/`across_speed` semantically consistent at any angle. Perpendicular drift (across_speed) always moves lines laterally regardless of rotation. Matches Vsynth x/y displacement vocabulary translated to the patch's own frame.

**Consequences:**
- `angle` is load-bearing — changing it rotates both the visual and the drift axes simultaneously
- World-space drift means the field is infinite; UV wrapping handles edges for free

---

### ADR 4: threshold dual semantics

**Context:** `threshold` controls density in source/density mode but has no obvious role in displacement mode.

**Decision:** In displacement mode, `threshold` scales the displacement amount — how far input luma warps the UV lookup. Same knob, context-dependent semantic. At `threshold = 0` in displacement mode: no warp.

**Rationale:** Economy of params. The knob is always active and always meaningful. The semantic shift is small enough to be intuitive from context.

**Consequences:**
- Label in UI is `threshold` — acceptable for both uses
- Documented in spec; no special codebox branching needed beyond the `proc_mode` switch already present

---

### ADR 5: Build script generates patcher JSON

**Context:** Consistent with f_lens, f_grain, f_masonry.

**Decision:** Python build script at `~/Vsynth/patterns/build_f_stipple.py`, reading committed patch as ground truth for structure. Writes to `patchers/f_stipple.maxpat`.

**Consequences:**
- Patcher is never hand-edited after build script exists
- Build script is a dev artifact, not in repo
- Build script written after codebox is confirmed working (codebox-first constraint)

## Implementation Phases

### Phase 0: Scratch patch setup
Create `stipple-scratch.maxpat` in `~/Vsynth/patterns/`. Minimal jit.gl.pix with codebox inlet for live editing. Wire to Vsynth context with a render trigger texture.

**Exit criteria:** Can paste codebox content and see output in Vsynth.

### Phase 1: Core hash field
Write codebox through threshold/softness. Verify Block 1. Note actual freq range and any seam behavior.

**Calibration tasks:**
- What freq value produces visually "default" grain? Set as param default.
- Does 2D hash break the diagonal seam? Document finding.

**Exit criteria:** Block 1 verification passes. Freq default and range noted.

### Phase 2: Drift + processor modes
Add along/across speed, time accumulation, source/processor auto-detect, proc_mode branch, bypass.

**Calibration tasks:**
- What speed value produces comfortable drift rate? Set ±1.0 scale accordingly.
- Decide processor/density compositing method (multiply vs luma mask vs mix).

**Exit criteria:** Block 2 verification passes. Speed scale and compositing method decided.

### Phase 3: Build script + patcher
Write `build_f_stipple.py`. Generate `f_stipple.maxpat`. Update freq range and speed range in build script from Phase 1/2 calibration findings.

**Exit criteria:** Block 3 verification passes.

### Phase 4: Integration + docs
Test in Vsynth signal chains (source mode, processor/density, processor/displacement). Write `docs/f_stipple.md`. Update `README.md` and build queue. Archive `.specify/f_stipple/` or leave for reference.

**Exit criteria:** Block 4 verification passes. README updated. `ideas/f_stipple.md` marked done.
