## Phase 4: Bombing (`bomb` param, K=2) — COMPLETE

**Note:** actual implementation diverged from this phase's original plan
in two ways, both resolved and documented in plan.md's ADR 8 addendum and
the toggle-behavior addendum:
- Stage 1 itself hit the same `DSL.Parser` compile ceiling as ADR 8 once
  extended to 18 candidates — required a further split into Stage 1a/1b
  (search halves, salt-parameterized reuse of one codebox) + Stage 1c
  (small merge). Pipeline is 6 stages, not 4.
- `sqrt(K)` density compensation was built, then removed (T017 below) —
  it made `bomb` read as a zoom/density control rather than a clustering
  control. `bomb`/`active_blend` also turned out to only work as an
  on/off toggle, not a graded fader — real priority magnitude scales with
  other params, so no fixed sentinel constant stays correctly scaled.
  Both are documentation corrections, not further code changes needed.

- [x] T016 Second sample per cell added — via Stage 1a/1b split
      (identical codebox, `salt1-4` params distinguish the two, rather
      than doubling candidates within one codebox)
- [x] T017 `sqrt(K)` density compensation implemented, then **removed**
      after empirical testing showed it caused a zoom-like read rather
      than a clustering read (`density_scale_eff` now equals `density_scale`
      unconditionally in both search halves) — see plan.md addendum
- [x] T018 `bomb=0` regression check — confirmed matching Phase 3's
      non-bombing output exactly (`active_blend` forces losing sentinel
      at 0, same "0 = identical to shipped" idiom used throughout)
- [x] T019 fps re-measured: 18-candidate search (6-stage chain,
      `bomb=1`) — **60+fps, confirmed 2026-07-04.** Cheap, not a concern.

**Checkpoint passed:** Full Evolution 2 mechanism (bombing + compositing)
proven in scratch, cost confirmed cheap at both `bomb=0` and `bomb=1`.

## Dependencies & Execution Order

**Phase dependencies**: 1 → 2 → 3 → 4 → 5, strictly sequential — each
phase's checkpoint is a precondition for starting the next, since Stage 2/3
depend on Stage 1's coord outputs existing and being correct, Stage 4
depends on Stage 2/3's render outputs, bombing (Phase 4) extends Stage 1
after the non-bombing chain is already proven end-to-end.

**Parallel opportunities**: T009 (second render instance) can happen

**Real fork point**: T015 — if Phase 3's fps number is unacceptable, do not

expensive chain). Return to plan.md's Complexity Notes / Phase 4.5
territory instead.

## Implementation Strategy

**Scratch-first, strictly staged.** Each phase is scratch-only until its
own checkpoint passes — no production file touched before Phase 5. This
mirrors both the general "codebox before patcher" constitution rule and the
specific lesson from this feature's first attempt (ADR 8): combining too
much into one untested codebox produced a failure mode (a Lua compile
ceiling) that had nothing to do with the actual math, and cost real time to
diagnose. Staging by pipeline stage keeps each scratch test small enough to
debug in isolation.

**Stop-and-reassess points**: end of Phase 1 (does Stage 1 alone compile
and behave correctly — this is the most direct re-test of ADR 8's fix),
end of Phase 3 (is the cost acceptable at all, before bombing adds more).
