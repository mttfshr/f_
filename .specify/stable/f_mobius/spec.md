# Spec: f_mobius

_Last updated: 2026-05-25_

## What it does

A UV-space processor that applies a Möbius transformation to the sampling coordinates of an incoming texture. Every pixel is remapped to a new source location — the texture itself is unchanged, but *where* each pixel samples from moves according to the transformation.

Möbius transformations map circles to circles (lines are circles of infinite radius) and preserve angles. The warp feels smooth and geometrically coherent even at extreme settings — no pinching, no shearing.

```
f(z) = (az + b) / (cz + d)
```

where z is the pixel coordinate as a complex number (x + iy). a, b, c, d are complex constants derived internally from the performer controls.

**Chain position**: Processor. Texture in → warped texture out. Sits between any source and any downstream processor or display layer. Composes cleanly with any other UV-space processor (e.g. f_droste).

**Acceptance criteria:**
- `invert=0, rotate=0, zoom=0.5` → pixel-perfect passthrough
- `invert=1, cx=0.5, cy=0.5` → clean inside-out inversion (center expands, edges contract)
- `invert=0, rotate=0.25` → 90° rotation of sampling coordinates
- cx/cy controllable via xy encoder (named float messages on in1)
- Bypass works; loads in Vsynth

## Parameters

| Param | Range | Default | Description |
|-------|-------|---------|-------------|
| `cx` | 0–1 | 0.5 | Center X — anchor of the transformation in UV space |
| `cy` | 0–1 | 0.5 | Center Y — anchor of the transformation in UV space |
| `rotate` | 0–1 | 0.0 | Rotation (mapped to 0–2π in codebox) |
| `zoom` | 0–1 | 0.5 | Scale — log-mapped: 0.5 = unity, 0 = 0.1×, 1 = 10× |
| `invert` | 0–1 | 0.0 | Inversion blend: 0 = rotation/zoom only, 1 = full 1/z |
| `bypass` | 0/1 | 0 | Standard bypass |

**cx / cy**: Intended for xy encoder. Standard named-message convention — `cx 0.4` and `cy 0.6` on in1. No special wiring needed beyond standard live.dial param_connect.

**invert**: The unique control. At 0: familiar rotation and zoom. At 1: pure inversion — inside and outside trade places. The interesting performance territory is **0.2–0.8**: partial inversion combined with rotation produces loxodromic (hyperbolic spiral) motion. Budget an exploration session here after basic build is confirmed.

**Edge behavior**: UV coordinates outside [0,1] after transformation are mirrored back in. Consistent with existing spherical-surface illusion approach.

## Codebox Structure

```
// Helpers
vec2 cmul(vec2 a, vec2 b)  // complex multiply
vec2 cdiv(vec2 a, vec2 b)  // complex divide

// Signal flow
1. Map norm → complex plane; offset by -(cx, cy)
2. Apply rotate + zoom → identity path result
3. Apply 1/z → inversion path result
4. mix(identity, inversion, invert)
5. Offset back by +(cx, cy); apply mirror edge guard
6. sample(in1, uv_out)
7. mix(effect_out, sample(in1, norm), bypass)
```

Separate identity and inversion paths (step 4) keeps the blend clean — no discontinuities as `invert` moves.

**Zoom mapping**: `pow(10.0, (zoom - 0.5) * 1.0)` — range 0.1× to 10×. Tune range empirically.

**Prefix**: `mob` (scripting name: `mob_pix`)

## Relation to Other Bpatchers

**Enables (build after f_mobius):**
- `f_stereo` — stereographic projection; sphere rotations in stereographic projection are Möbius transformations
- `f_poincare` — Poincaré disk; disk automorphisms are the Möbius transformations that keep the unit disk inside itself

**Shares math with:**
- Apollonian gasket — generated by repeated circle inversions; the inversion primitive in f_mobius is the same operation
- Non-Euclidean / hyperbolic geometry generally

**Works well with:**
- `f_chladni` — Möbius-warped Chladni patterns; nodal rings stay rings, layout inverts and rotates
- `f_droste` — both are UV-space processors; both orders interesting
- Any radially-symmetric source — circle-preserving property respects radial symmetry

## Open Questions

- **Zoom range**: 0.1–10× is a reasonable starting point; tune empirically for what's performable
- **Loxodromic sweet spot**: invert 0.2–0.8 + rotate is the most interesting territory; needs a performance exploration session to find expressive range
- **Second xy encoder**: could rotate + zoom also be driven by a second encoder (rotate on x, zoom on y)? Decide at build time.
- **Singularity guard**: where denominator → 0, NaN/inf may escape. Verify mirror edge guard catches all cases; add explicit clamp if needed.
