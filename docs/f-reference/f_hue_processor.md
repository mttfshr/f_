# f_hue_processor ‚Äî Bpatcher Spec

_Last updated: 2026-07-05 (previous doc dated 2026-05-25 described a superseded rslider-based implementation ‚Äî see Loose Threads)_
_Status: Working, with a known open UI bug (see Loose Threads)_

## Concept

Hue-selective processing. Isolates a flat-topped hue band ‚Äî centered on `hue_center`, extending `hue_lower` degrees below and `hue_upper` degrees above, with a `edge_falloff`-degree smoothstep at each boundary ‚Äî and applies saturation, luminance, and hue-rotation adjustments only within that band. A low-saturation gate (`smoothstep(0.05, 0.15, S)`) suppresses the effect on near-grey pixels regardless of hue, avoiding hue-selection noise in desaturated regions.

## Parameters

| Name | Type | Range | Default | Description |
|------|------|-------|---------|-------------|
| `hue_center` | float | 0‚Äì360 | 120.0 | Hue band center, in degrees |
| `hue_lower` | float | 0‚Äì180 | 36.0 | Flat-top extent below center, in degrees |
| `hue_upper` | float | 0‚Äì180 | 36.0 | Flat-top extent above center, in degrees |
| `edge_falloff` | float | 0‚Äì90 | 10.0 | Smoothstep falloff width at each boundary, in degrees ‚Äî 0=hard edge |
| `sat_amt` | float | -1‚Äì1 | 0.0 | -1=full desaturate, 0=unchanged, 1=full boost, within the band |
| `lum_shift` | float | -1‚Äì1 | 0.0 | Additive luminance shift within the band |
| `hue_shift` | float | -180‚Äì180 | 0.0 | Hue rotation within the band, in degrees |
| `bypass` | toggle | 0/1 | 0 | Bypass processing |

## Signal Chain

## Loose Threads

- **HANDOFF.md (as of the last session touching this module) still lists an open bug: "f_hue_processor bugs (hue_lower/hue_upper missing UI path)."** The previous version of this doc (dated 2026-05-25) attributed this to `hue_lower`/`hue_upper` being `rslider` widgets deliberately excluded from `route` because the rslider couldn't be driven remotely. **That no longer matches the shipped `definition.py` (last updated 2026-06-15)** ‚Äî `hue_lower`/`hue_upper` are now plain floats in the same params list as everything else, with no special-casing visible in `definition.py`. Either the rslider-based implementation described in the old doc predates a rewrite to the current flat-top/falloff model and the "missing UI path" bug is a *new*, differently-caused issue in the current build, or the bug note in HANDOFF is itself stale. **Needs a fresh look in Max to characterize the actual current symptom** ‚Äî this doc can't resolve which is true from the files alone.
- The underlying hue model changed from whatever the 2026-05-25 doc described to the current `hue_center` + `hue_lower`/`hue_upper` (asymmetric flat-top) + `edge_falloff` model ‚Äî worth being aware of if cross-referencing any older notes or muscle memory about this module's dial layout.
